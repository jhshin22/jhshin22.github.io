## Module Summary

- `constants.js` defines piece types, counts, labels, and generated canvas icons.
- `state.js` creates `A.S`, the shared mutable game state.
- `movement.js` exposes `A.legal`, `A.findMove`, and `A.inBounds`.
- `battle.js` exposes `A.battleRank`.
- `victory.js` exposes helper predicates and `A.checkEnd`.
- `setup.js` creates piece pools, resets state, manages setup zones, and fills random placement.
- `render.js` renders the board, tray, side panel, cover modal, and alert modal.
- `main.js` drives setup, move application, combat application, pass, turn changes, and boot.
- `ai.js` provides base AI mode detection, move enumeration, scoring, and scheduling.
- `ai_setup.js` overrides setup with named strategic plans for random, spring, and summer modes.
- `ai_summer.js` wraps AI scheduling for summer mode and adds memory-based scoring.
- `ai_special_guard.js` filters wasteful non-attack special moves.
- `ai_policy_guard.js` wraps `A.legal`, `A.scheduleAiTurn`, `A.applyMove`, and `A.resetAi` for summer policy constraints.
- `ai_endgame_guard.js` wraps `A.scheduleAiTurn` and `A.resetAi` for forced endgame attacks.
- `match_log.js` wraps move, turn, alert, and reset behavior to record browser logs.

## Execution Flow

1. `DOMContentLoaded` calls `boot()` in `main.js`.
2. `A.reset()` initializes setup state and piece pools.
3. Bottom player places 16 pieces, then finishes setup.
4. In AI modes, top setup is generated by `A.aiSetup`; otherwise top player places pieces manually.
5. Playing starts with bottom turn.
6. Human clicks select a piece, `A.legal` generates legal moves, then `A.applyMove` applies a selected move.
7. Combat calls `A.battleRank`, reveals involved pieces, removes captured pieces, and may show the alert modal.
8. `A.endTurn` checks victory/draw, switches turn, and schedules AI if the top player is controlled by AI.
9. AI wrappers may filter legal moves, force emergency actions, or call the previous scheduler.
10. `match_log.js` records moves and finishes logs when game-over alerts complete.

## Log Structure

Logs are stored in browser `localStorage` under `chuWar.matchLogs.v1`, capped at 30 entries. A log includes mode, start/end time, initial board, move records, before/after board snapshots, remaining piece counts, result, final board, and visible text logs.

## GitHub Pages Conditions

The game uses relative paths under `Chu_war/`, so it is compatible with `https://jhshin22.github.io/Chu_war/`. It has no required package installation or build output. Browser APIs used include DOM, canvas, `localStorage`, clipboard, `Blob`, object URLs, and timers.

## Known Risks

- Many core files are compressed into a single line, making review and small diffs harder.
- Game state, rendering, AI, alerts, timers, and logging all share the global `window.ChuWar` object.
- Several files wrap the same functions in load order; changing script order can alter behavior.
- `match_log.js` records true hidden piece identities for analysis, so simulation code must avoid feeding those records back into gameplay AI.
- Direct `document`, `localStorage`, `navigator.clipboard`, `setTimeout`, and modal dependencies must be adapted before headless simulation.
